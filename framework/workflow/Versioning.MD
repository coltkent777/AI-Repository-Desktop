# Purpose

This document defines the versioning standards for the Art Framework and all associated documentation.

Versioning exists to provide a complete history of changes, preserve canon, and make every modification traceable.

---

# Versioning Philosophy

Every permanent document should maintain its own version history.

Framework, campaigns, characters, and artwork evolve independently.

A change to one document should not require version changes to unrelated documents.

---

# Documents Requiring Versions

The following maintain independent version numbers:

- Framework documents
- Campaign documents
- Character documents
- Artwork
- Templates

Each document is responsible for maintaining its own revision history.

---

# Version Format

Semantic Versioning (SemVer) is used.

MAJOR.MINOR.PATCH

Example:

1.0.0

---

# Version Definitions

## Patch Release (X.X.1)

Used for:

- Typographical corrections
- Formatting improvements
- Clarifications
- Documentation cleanup

Patch releases do not modify canon.

---

## Minor Release (X.1.0)

Used for:

- Approved canon additions
- New documented abilities
- Additional sections
- Expanded documentation
- New approved artwork standards

Minor releases add information while maintaining compatibility.

---

## Major Release (2.0.0)

Used for:

- Framework restructuring
- Canon rewrites
- Breaking documentation changes
- Rule modifications
- Significant organizational changes

Major releases should be uncommon.

---

# Artwork Versioning

Individual artwork should also maintain version numbers.

Examples include:

- Character Portrait v1
- Character Portrait v2
- Logo v3
- Collector Card v4

Artwork revisions allow visual improvements without modifying canon.

---

# Canon and Versioning

Artwork revisions do not automatically require documentation version changes.

Documentation versions should only change when canon or documentation itself changes.

Visual improvements alone are not canon updates.

---

# Revision History

Every version change should include a brief summary explaining what changed.

Examples include:

- Added signature familiar.
- Updated magical language.
- Refined ornamentation.
- Corrected documentation.
- Clarified rendering standards.

Revision history should describe the reason for the update rather than simply stating that a change occurred.

---

# Change Approval

Version numbers should only change after approved modifications.

Recommendations, brainstorming, and discussion do not create new versions.

Only accepted changes require a version increment.

---

# Framework Integrity

Version history serves as the permanent record of framework evolution.

No approved change should occur without an accompanying version update and revision history entry.
